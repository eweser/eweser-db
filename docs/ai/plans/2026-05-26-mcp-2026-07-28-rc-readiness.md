# Plan: MCP 2026-07-28 RC Readiness

Status: draft, awaiting approval

## Goal

Bring EweserDB's remote MCP endpoint, OAuth connection flow, MCP tool contracts,
and Connect AI guidance into readiness for the MCP `2026-07-28` release while
preserving current `2025-11-25` client compatibility and room-scoped access
controls.

## Scope

- In:
  - `packages/auth-server-hono` remote MCP `/mcp` transport behavior,
    stateless request handling, route tests, CORS/header behavior, OAuth
    metadata, dynamic client registration, token issuance, and migrations if
    OAuth schema changes are required.
  - `packages/mcp-server` tool registration metadata, structured tool outputs,
    output schemas, deterministic list behavior, secret redaction, package
    tests, and README updates.
  - `packages/app` Connect AI copy/settings and tests only where the protocol
    and OAuth changes alter user-visible setup guidance.
  - MCP smoke scripts/docs needed to verify both legacy and `2026-07-28`
    clients against the local stack.
  - Current docs that describe MCP setup, security, deployment headers, or
    client compatibility.
- Out:
  - Implementing MCP Apps UI components.
  - Implementing the Tasks extension unless Run 1 proves a target client needs
    it for launch compatibility.
  - Building a local encrypted-room bridge or key-delegation model.
  - Replacing Hocuspocus/Yjs room sync, changing room document schemas, or
    changing public aggregation behavior.
  - Generic third-party OAuth marketplace support beyond the supported Connect
    AI clients.
  - Removing legacy `2025-11-25` compatibility before target clients have moved.

## Assumptions / Open Questions

- Assumption: The MCP release candidate was locked on May 21, 2026, and final
  publication is planned for July 28, 2026.
- Assumption: The `2026-07-28` RC is a breaking protocol update. The main
  relevant changes are stateless requests, no protocol-level `Mcp-Session-Id`,
  no `initialize` handshake, `server/discover`, required `Mcp-Method` and
  `Mcp-Name` headers for Streamable HTTP POSTs, cache metadata for list/read
  results, first-class extensions, OAuth hardening, and full JSON Schema
  2020-12 tool schemas.
- Assumption: As of this plan, local `@modelcontextprotocol/sdk` is `1.29.0`
  and reports `LATEST_PROTOCOL_VERSION: "2025-11-25"`. Do not hand-roll a full
  RC transport unless Run 1 proves official SDK support is unavailable and the
  user approves that risk.
- Assumption: Current Eweser remote MCP uses a session cache keyed by
  `mcp-session-id` in `packages/auth-server-hono/src/routes/mcp.ts`; this must
  not remain a protocol requirement for RC clients.
- Assumption: Current OAuth metadata exposes `issuer`, but authorization
  redirects do not add an `iss` parameter, dynamic registration does not model
  `application_type`, and OAuth access tokens do not yet have refresh-token
  support.
- Open question: Which target clients will support `2026-07-28` first, and do
  they require strict RC-only behavior or tolerate dual legacy/RC endpoints?
- Open question: Should the production endpoint stay one URL with negotiated
  compatibility, or should Eweser expose a temporary RC validation URL such as
  `/mcp/rc` until the final spec ships?
- Open question: Do Claude web, ChatGPT web, Codex, Copilot, or OpenClaw require
  OAuth refresh tokens for acceptable user experience, or are short-lived access
  tokens plus reconnect enough for the first RC pass?

### Run 3: Make Remote MCP Transport Stateless-Compatible

- **Id**: `run-3`

- **Deliverable**:

  - `packages/auth-server-hono/src/routes/mcp.test.ts`: legacy and RC route

  - `packages/auth-server-hono/package.json` and root lockfile: upgrade

  - `scripts/test-mcp.mjs` and/or `scripts/test-mcp2.mjs`: add RC request
    smoke coverage or split into a new script if clearer.
- **Steps**:

        or stop for approval before implementing a custom transport shim.

  - [ ] Confirm rate limiting can use method/name headers instead of body

## Stop Conditions

Stop and ask for user approval if:

- Official `@modelcontextprotocol/sdk` support for `2026-07-28` is unavailable
  and implementation would require a bespoke full transport or JSON-RPC stack.
- The final July 28, 2026 spec differs materially from the May 21, 2026 RC.
- A target client requires a different OAuth model than this plan covers, such
  as confidential-client credentials, private-key JWT, or provider-specific
  registration.
- Implementation needs to remove legacy `2025-11-25` compatibility before
  Claude/ChatGPT/Codex/OpenClaw compatibility is proven.
- A PostgreSQL migration is needed outside OAuth operational metadata.
- A published package API or runtime contract changes without a planned
  changeset.
- Encrypted-room, MCP Apps, or Tasks-extension product scope becomes necessary
  for launch.
- Required secrets, production credentials, or external provider dashboards are
  needed to verify behavior.
